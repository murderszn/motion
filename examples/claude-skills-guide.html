a field guide to claude code context

Skills, CLAUDE.md,
and what makes
them different.

Claude Code has several ways to feed it knowledge — and they are not interchangeable. This is the practical map: what CLAUDE.md is for, how Agent Skills work, where AGENTS.md fits, and why subagents are a different animal entirely.

01  always-on memory 02  on-demand skills 03  the open standard 04  isolated subagents
tl;dr

One axis explains almost everything: always loaded vs. loaded when needed.

Every mechanism below is a way to put the right knowledge in front of Claude at the right time. The trick is understanding when each one enters the context window — because context is finite, and what's loaded shapes every response.

persistent context

CLAUDE.md & AGENTS.md

Read automatically at the start of a session and kept in context the whole time. Good for the handful of facts that are always relevant — conventions, commands, architecture.

progressive disclosure

Agent Skills

Only a name + one-line description sit in context. The full instructions and bundled files load only when Claude judges the skill relevant. Cheap to have many.

cross-vendor standard

AGENTS.md

A tool-neutral instructions file many agents read. Claude Code doesn't read it natively — you bridge it into CLAUDE.md with a one-line import.

isolated worker

Subagents

A separate Claude with its own context window, system prompt, and tools. It does a job and reports back a summary — it doesn't pollute your main thread.

01 — project memory

CLAUDE.md

CLAUDE.md is Claude Code's memory file. When a session starts, Claude reads it automatically and keeps it in context for the entire conversation. It's the place for the things you'd otherwise repeat in every prompt.

What belongs in it

  • Commands — how to build, test, lint, and run the project
  • Conventions — naming, file structure, style rules, branching
  • Architecture — the mental model: where things live and why
  • Gotchas — non-obvious constraints that aren't visible in the code

Keep it tight. Everything here costs context tokens on every turn, so it should be high-value and broadly relevant — not a dumping ground.

Where it lives

Claude Code merges CLAUDE.md from several locations, most-specific winning:

claude.md resolution order
# 1. user-global — applies to every project you touch
~/.claude/CLAUDE.md

# 2. project root — committed, shared with the team
./CLAUDE.md          # (or ./.claude/CLAUDE.md)

# 3. project-local — your private overrides, gitignored
./CLAUDE.local.md

# 4. subdirectory — lazy-loaded only when you work in that folder
./packages/api/CLAUDE.md

It can import other files

Use @path imports to pull in other files so CLAUDE.md stays a thin index pointing at deeper docs:

CLAUDE.md
# Project: Acme API

See @docs/architecture.md for the system overview.
Coding standards live in @docs/conventions.md.

## Commands
- build: npm run build
- test:  npm test -- --watch=false

Bootstrap a starter file at any time by running /init — Claude scans the repo and drafts a CLAUDE.md for you (and it even reads an existing AGENTS.md while doing so).

02 — agent skills

Skills

A Skill is a folder with a SKILL.md file that packages expertise for a specific kind of task — a workflow, a domain, a tool integration. The defining trait is progressive disclosure: Claude only sees the skill's name and description until it decides the skill is relevant, then it loads the full body and any bundled files on demand.

That's the whole point. You can install dozens of skills and pay almost nothing in context for the ones you don't use — unlike CLAUDE.md, which is always loaded in full.

Anatomy of a skill

.claude/skills/pdf-forms/
pdf-forms/
├── SKILL.md          # required: frontmatter + instructions
├── reference.md      # optional: loaded only when needed
└── scripts/
    └── fill.py       # optional: bundled code Claude can run

The SKILL.md frontmatter

SKILL.md
---
name: pdf-forms
description: Fill, flatten, and extract fields from PDF
  forms. Use when the user mentions PDFs, form fields,
  or acroform data.
disable-model-invocation: false   # auto-trigger on match
run-in-subagent: false            # run in main thread
---

# PDF Forms

Step-by-step instructions go here. Reference bundled
files like reference.md — they load only when Claude
opens them, keeping the base footprint tiny.

The description is the single most important line you write: it's the only thing Claude sees at rest, so it's what decides whether the skill ever triggers. Write it in terms of when to use this, not just what it is.

Where skills live

  • .claude/skills/ — project skills, committed and shared with the team
  • ~/.claude/skills/ — personal skills, available across all your projects
  • Plugins — skills can ship inside a plugin and be distributed to others

Skills are also portable: the same SKILL.md format works across the Claude apps, Claude Code, and the API (via code execution). Write once, use everywhere.

03 — the open standard

AGENTS.md

AGENTS.md is a vendor-neutral standard: a plain-markdown instructions file that many different coding agents agree to read. The goal is one file your whole toolchain can share, instead of a different config for every tool.

Here's the catch people miss: Claude Code does not read AGENTS.md natively. It reads CLAUDE.md. So if your repo already has an AGENTS.md, the clean move is to import it:

CLAUDE.md
# Single source of truth for every agent
@AGENTS.md

Now both Claude Code and any AGENTS.md-aware tool read the same instructions, and you maintain one file. Two other things worth knowing:

  • /init reads it. When you bootstrap CLAUDE.md, Claude pulls in relevant parts of an existing AGENTS.md (and similar files like .cursorrules) automatically.
  • Precedence is simple. For Claude Code, CLAUDE.md is authoritative — an imported AGENTS.md becomes part of CLAUDE.md's content.

Conceptually AGENTS.md is the same category as CLAUDE.md — always-on persistent instructions. It differs only in who reads it: AGENTS.md is cross-tool, CLAUDE.md is Claude-specific.

04 — side by side

How they differ

The same four mechanisms across the dimensions that actually matter:

05 — the different animal

Why subagents aren't skills

This is the comparison people get wrong most often. A skill is knowledge that loads into the conversation you're already having. A subagent is a whole separate Claude you hand a task to — it has its own context window, its own system prompt, and its own tool permissions, and it returns only a summary.

skill

Augments the current thread

Adds instructions and files to the context you're in. Claude keeps doing the work itself, just better-informed. No isolation — everything stays in your one conversation.

subagent

Delegates to a fresh context

Spins up an isolated worker for a chunk of work — searching a huge codebase, parsing logs — so the noise never touches your main window. You get back the conclusion, not the mess.

Rule of thumb: reach for a subagent when a task would flood your context with intermediate junk, or needs different tool constraints. Reach for a skill when you want Claude to know how to do something well, in the flow of the current conversation. They compose — a skill can even be configured to run-in-subagent.

06 — choosing

Which one do I reach for?

always true for this repo?
Commands, conventions, architecture that apply to every task → CLAUDE.md.
shared across many tools?
Want Cursor, Codex, and Claude to read one file → write AGENTS.md and import it from CLAUDE.md.
a repeatable specialized task?
A workflow that only matters sometimes, possibly with scripts → package a Skill.
a big, noisy, isolatable job?
Heavy search, log parsing, or work needing its own tools → delegate to a Subagent.

In practice you use all four together: CLAUDE.md sets the ground rules, Skills teach specialized moves, Subagents handle the heavy lifting, and AGENTS.md keeps the rest of your toolchain in sync.
